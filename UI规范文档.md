# 《会来事儿》UI 规范文档

> 文档版本：v1.0  
> 更新时间：2026-05-26  
> 适用范围：《会来事儿》MVP 电脑端 Web Demo  
> 对应需求文档：《会来事儿完整产品需求文档.md》  
> 首个场景：初入职场 / 以茶代酒  
> 核心 UI 方向：动画风格、轻松、易操作

## 1. 设计目标

### 1.1 产品气质

《会来事儿》的 UI 应该像一部可以互动的轻松职场动画短剧，而不是严肃培训系统、考试软件或写实饭局模拟器。

整体感受：

- 轻松，不压迫
- 动画感强，但不幼稚
- 职场感明确，但不沉重
- 社交压力可感知，但不让用户紧张到不想操作
- 学习反馈清楚，但不显得像扣分考试

### 1.2 设计关键词

```text
轻松
动画感
易操作
有分寸
有温度
可复盘
低压力学习
```

### 1.3 体验目标

用户进入页面后，需要快速理解三件事：

```text
我是谁
现在是什么局面
我要说一句什么话
```

UI 要帮助用户观察场景，而不是让用户在复杂控件里迷路。

## 2. 视觉风格

### 2.1 总体风格

采用 **2D 动画插画风**。

推荐方向：

- 扁平插画 + 轻微立体阴影
- 无外框动画人物头肩像或半身像
- 表情夸张但克制
- 圆桌、菜品、茶杯、酒杯用图形化表达
- 包间氛围温暖，但不写实油腻
- 界面像互动剧情游戏，不像后台管理系统

避免方向：

- 写实照片风
- 古风江湖风
- 纯聊天软件风
- 严肃企业培训风
- 过度卡通低幼风
- 酒桌文化宣传感
- 大量红色警告造成压迫

### 2.2 画面基调

首个场景「以茶代酒」发生在客户饭局包间，但视觉不能过度还原真实酒桌的压迫感。

推荐表达：

```text
暖色包间灯光
+ 柔和圆桌
+ 动画人物头肩像/半身像
+ 清晰对话气泡
+ 轻量状态反馈
```

### 2.3 图形语言

- 人物使用无外框动画头肩像或半身像，直接融入酒桌场景；不使用圆形头像框、圆圈边框或卡片式头像框。
- 五官用简单几何图形表达。
- 角色情绪通过眉毛、嘴型、脸色、动作小幅变化体现。
- 酒桌使用俯视或轻微斜视角，不使用真实照片。
- 茶杯、酒杯、菜品要能被识别，但不需要复杂细节。
- 状态条使用柔和颜色和动画变化，不使用生硬的考试分数样式。

## 3. 颜色规范

### 3.1 主色

| 用途 | 色值 | 说明 |
| --- | --- | --- |
| 主品牌色 | `#23645B` | 稳重、成熟，用于主按钮和重点标识 |
| 辅助青绿 | `#187781` | 用于信息提示、状态强调 |
| 温暖米白 | `#F6F2E8` | 页面背景，降低紧张感 |
| 纸张白 | `#FFFAF0` | 面板和卡片背景 |
| 墨色文字 | `#1D2525` | 主文字 |

### 3.2 辅助色

| 用途 | 色值 | 说明 |
| --- | --- | --- |
| 面子 / 气氛 | `#C4812D` | 温暖、社交氛围 |
| 边界 / 正向 | `#1F8A5B` | 稳定、健康、正向 |
| 风险 | `#C64C3C` | 用于风险提示，不大面积使用 |
| 轻提示底色 | `#E8F0EB` | 标签、知识卡、正向反馈 |
| 分隔线 | `rgba(65, 71, 67, 0.16)` | 轻量结构分隔 |

### 3.3 颜色使用原则

- 页面整体不能被单一色系统治。
- 背景以米白和暖光为主。
- 主按钮统一使用主品牌色。
- 风险红只用于风险状态、负向变化、危险提醒。
- 正向反馈使用绿色，但不要让页面变成全绿色。
- 状态条颜色要有区分度，用户一眼能看出风险和正向状态。

## 4. 字体与排版

### 4.1 字体

推荐字体栈：

```css
font-family:
  Inter,
  "Microsoft YaHei",
  "PingFang SC",
  "Noto Sans CJK SC",
  system-ui,
  sans-serif;
```

### 4.2 字号

| 场景 | 字号 |
| --- | --- |
| 首页产品名 | 48-58px |
| 页面标题 | 28-40px |
| 面板标题 | 18-22px |
| 正文 | 15-17px |
| 辅助说明 | 12-14px |
| 状态文字标签 | 13-15px |

桌面网页字号：

- 页面标题不小于 28px。
- 正文不小于 15px。
- 输入框文字不小于 16px，保证长时间输入舒适。

### 4.3 排版原则

- 不使用负字距。
- 不使用随视口宽度无限缩放的字体。
- 长句必须自动换行，不得横向溢出。
- 对话文本行高建议 `1.55-1.75`。
- 反馈和复盘内容要分段，不堆成大段文字。

## 5. 布局规范

### 5.1 页面宽度

桌面 Web：

```text
内容最大宽度：1180px
页面左右边距：28px
主内容居中
```

### 5.2 圆角与阴影

- UI 面板、按钮、输入框统一使用 `8px` 圆角。
- 酒桌等场景物件可以使用圆形；人物不放入圆形或圆角头像容器。
- 阴影要轻，不做厚重悬浮卡片堆叠。
- 禁止卡片套卡片的复杂视觉层级。

### 5.3 间距

| 级别 | 间距 |
| --- | --- |
| 页面大区块间距 | 24-32px |
| 面板间距 | 16px |
| 面板内边距 | 18-24px |
| 按钮和输入区间距 | 12-16px |
| 标签间距 | 8-10px |

## 6. 页面 UI 规范

### 6.1 首页

目标：

- 让用户知道这是一款 AI 人情世故训练游戏。
- 让用户快速开始，不做复杂营销页。

必须包含：

- 产品名：会来事儿
- 说明文案：在真实人情场景里，练习有分寸地表达。
- 主按钮：开始游戏
- 动画化酒桌预览图

首页结构：

```text
左侧：产品名、定位、开始按钮
右侧：动画化圆桌预览
```

首页动效：

- 圆桌预览轻微浮动。
- 头像轻微呼吸动效。
- 主按钮 hover 时轻微上浮。
- 页面加载时内容淡入，不超过 300ms。

### 6.2 场景介绍页

目标：

- 让用户知道当前身份、场景、冲突和学习点。

必须包含：

- 章节：初入职场
- 场景名：以茶代酒
- 玩家身份：林一，入职三个月的项目助理
- 场景冲突：客户饭局被劝酒
- 学习点标签：面子、分寸、潜台词、职场关系、边界与风险
- 按钮：进入酒桌

设计要求：

- 文案简短，避免一上来大段说明。
- 学习点用轻量标签展示。
- 按钮位置明显，用户一眼知道下一步。

### 6.3 游戏主界面

游戏主界面是 MVP 最重要的页面。

必须包含：

```text
顶部信息栏
顶部当前局势滚动字幕
中部动画酒桌场景
剧情提示文本
当前回合目标
左下角信息抽屉组
底部自由输入区
```

#### 6.3.1 顶部信息栏

展示：

```text
会来事儿
初入职场 / 以茶代酒
第 N 轮
当前目标：让客户接受边界
```

要求：

- 当前轮次用小徽章显示，不写死为 `第 1/2 轮`。
- 回合目标用文字表达，例如“说清边界”“给出替代动作”“把心意落到合作上”。
- 回合目标是方向提示，不是标准答案。
- 不要占用过多高度。
- 窗口缩窄时可在顶部信息栏内自然换行。

#### 6.3.1.1 当前局势滚动字幕

刚进入游戏主界面时，当前局势不直接以常驻面板展示，而是先以顶部滚动字幕形式出现。

展示内容：

```text
场合：客户庆功饭局
压力来源：李总第一次敬酒
你的事实：今晚开车、酒量很差、明天上午复盘
隐性目标：拒酒但不让客户当场下不来台
```

视觉要求：

- 字幕位于顶部信息栏下方或顶部浮层区域。
- 使用轻量半透明纸张底色，不能遮挡主要人物和酒桌。
- 文案以横向滚动、逐条滑入或字幕条方式输出。
- 字幕播放完成后自动收起。
- 推荐播放时长 4-6 秒；播放完成后不再占用顶部空间。
- 用户重新 hover 或点击左下角“当前局势”标签时，可再次查看完整内容。

动效要求：

- 字幕滚动速度要慢，保证可读。
- 不使用跑马灯式高速循环。
- 字幕只自动播放一遍，不无限循环。
- 开启减少动态模式时，改为静态显示 4 秒后收起。

#### 6.3.1.2 李总开场动态台词字幕

进入游戏主界面后，李总需要先说出第一句劝酒台词。该台词不能只是静态文案，需要用轻量字幕或气泡动效让用户感到“现场开始了”。

台词内容：

```text
林一是吧？年轻人第一次见，来，咱俩喝一个。
```

视觉要求：

- 台词优先使用李总头顶动态字幕或对话气泡，不使用远离角色的聊天列表。
- 字幕或气泡需要与李总人物位置产生明确关联。
- 台词出现时，李总人物同步播放看向林一、端杯、微笑、身体微微前倾的轻量动效。
- 台词播放完成后，输入区获得视觉焦点，提示用户自由输入。
- 不出现 A/B/C/D 或任何预设回复按钮。

动效要求：

- 推荐时长 1.8-2.8 秒。
- 可采用逐字显现、短句滑入、气泡轻弹入。
- 不使用高速跑马灯，不让用户追着文字读。
- 开启减少动态模式时，台词静态展示 2 秒后进入可输入状态。

#### 6.3.1.3 角色头顶对话字幕

游戏主界面的对话应该像动画短剧中的现场交流。所有角色说话都在对应人物头顶展示，避免界面变成传统聊天窗口。

适用对象：

```text
李总
王经理
周姐
阿诚
你：林一
```

展示规则：

- 用户点击“发送回应”后，输入内容先显示在林一头顶。
- AI NPC 回应时，台词显示在对应 NPC 头顶。
- 同一时间最多强调 1 条主台词，其他台词可以淡出或收起。
- 头顶字幕需要跟随角色位置，不能漂到无关区域。
- 长句需要自动换行，必要时拆成 2-3 个短句依次展示。
- 头顶字幕不得遮挡其他角色的脸、姓名、关系状态文字或底部输入框。
- 当前说话角色可以轻微高亮，帮助用户知道是谁在说话。
- 被说话对象要通过视线或身体朝向体现；李总对林一说话时，必须看向林一。

交互顺序：

```text
玩家输入
→ 林一头顶字幕出现
→ 林一字幕淡出或弱化
→ NPC 头顶字幕逐条出现
→ 当前回合目标更新
→ 状态反馈和点评出现
```

#### 6.3.2 动画酒桌场景

视觉必须包含：

- 包间暖光背景
- 圆形酒桌
- 菜品
- 茶杯
- 酒杯
- 角色头像
- 所有角色的头顶动态字幕或对话气泡

角色位置：

```text
├─ 左上：王经理
├─ 右上：李总
├─ 左下：周姐
├─ 右下：阿诚
└─ 下方：玩家林一
```

#### 6.3.3 角色头像规范

每个角色头像需要包含：

- 头像
- 姓名
- 身份
- 剧情定位短语
- 关系状态文字
- 当前状态短语
- 表情动效状态

示例：

```text
李总
客户负责人 · 压力发起者
仍在观察
端着酒杯
```

角色视觉差异：

| 角色 | 剧情定位短语 | 视觉特点 |
| --- | --- | --- |
| 李总 | 压力发起者 / 最终认可者 | 端酒杯、看向林一、表情热络但略有压迫感 |
| 王经理 | 直属领导 / 观察者 | 观察局面、表情克制 |
| 周姐 | 资深同事 / 气氛缓冲 | 关注气氛、表情温和 |
| 阿诚 | 同事 / 跟风放大器 | 跟着热闹、略显随意 |
| 林一 | 你 / 新人助理 | 处于下方视角位，显示“你：林一” |

角色定位展示要求：

- 定位短语必须短，建议 4-10 个字，避免挤压人物区域。
- 定位短语用于帮助用户理解每个人在场景里的作用，不显示数值。
- 李总的“最终认可者”定位需要在通关前保持可见或可通过 hover 查看。
- 桌面空间不足时，可只展示姓名和定位短语，身份放入 hover 详情。

情绪状态建议：

```text
中性
施压
缓和
认可
尴尬
担心
打圆场
```

表情动效要求：

| 情绪状态 | 表情与动作表现 |
| --- | --- |
| 中性 | 轻微呼吸、自然眨眼 |
| 施压 | 眉毛轻挑、端杯前递、身体微微前倾 |
| 缓和 | 嘴角放松、酒杯放低、点头 |
| 认可 | 微笑、点头、视线转向玩家 |
| 尴尬 | 视线轻微偏移、嘴角收住 |
| 担心 | 眉头轻皱、动作变慢 |
| 打圆场 | 手势轻摆、侧身接话、表情温和 |

设计限制：

- 表情动效必须服务于情绪理解，不做夸张表演。
- 人物仍然使用无外框动画头肩像或半身像，不使用圆形头像框、圆圈边框或卡片式头像框。
- 角色动效不能遮挡姓名、身份、关系状态文字或台词。
- 角色说话时需要有明确视线方向；李总对林一说话时，视线和身体朝向必须指向林一。

#### 6.3.4 左下角信息抽屉组

游戏主界面不常驻展示大面积“当前局势”和“状态反馈”面板。它们应收纳在页面左下角的信息抽屉组中，默认隐藏，用户需要时再查看。

信息抽屉包含三个入口：

```text
当前局势
状态反馈
小贴士
```

默认状态：

- 仅露出小标签或图标+文字标签。
- 固定在页面左下方。
- 不遮挡自由输入框和主要酒桌人物。
- 鼠标 hover 展开。
- 键盘 focus 展开。

展开状态：

- 展开轻量面板，显示对应内容。
- 鼠标移出或点击空白处后收起。
- 同一时间最多展开一个抽屉内容。
- 展开层不得遮挡用户正在输入的文本框。

#### 6.3.4.1 当前局势抽屉

展示四项：

```text
场合：客户庆功饭局
压力来源：李总第一次敬酒 / 李总二次施压
你的事实：今晚开车、酒量很差、明天上午复盘
隐性目标：拒酒但不让客户当场下不来台
```

设计要求：

- 顶部滚动字幕播放完成后，当前局势收起到左下角“当前局势”标签。
- hover 或 focus 后完整展示四项内容。
- 用 `dt/dd` 或明显标签结构。
- 不使用大段解释。

#### 6.3.4.2 状态反馈抽屉

展示：

```text
气氛
面子
边界
信任
风险
```

状态条要求：

- 状态条可以使用内部 0-100 数值控制长度，但前端不直接展示具体数字。
- 状态变化时有 200-400ms 的平滑动画。
- 状态条旁必须显示文字等级，例如“气氛融洽”“边界清晰”“风险较低”。
- 状态变化标签必须使用文字，例如“气氛更融洽”“风险降低”，不显示 `+8`、`-18`。
- 风险使用红色，越高越需要警觉。
- 其他状态使用青绿、绿色、暖色区分。
- 不要用“扣分感”很强的设计。
- 默认隐藏，仅显示“状态反馈”标签。
- hover 或 focus 后展开。
- 展开内容应简短，优先显示文字等级和状态条，不展示解释长文。

#### 6.3.4.3 小贴士抽屉

小贴士用于给用户低压力提示，不应像答案提示或标准答案按钮。

内容示例：

```text
先承接好意，再说明边界。
给出替代动作，例如以茶代酒。
别把压力直接甩给领导。
```

展示规则：

- 默认隐藏，仅显示“小贴士”标签。
- hover 或 focus 后展开。
- 每次最多展示 1-3 条短提示。
- 不显示可点击选项，不提供一键填入。
- 文字语气轻松，像旁白提醒，不像考试提示。
- 每轮提交后，小贴士根据 AI 评估更新，引导用户下一轮补齐缺失表达。

多轮引导示例：

| 当前缺失 | 小贴士文案示例 |
| --- | --- |
| 缺少边界 | “先把不能喝的事实说清楚。” |
| 缺少替代 | “可以给一个不喝酒也有礼数的动作。” |
| 缺少面子 | “先接住对方的好意，场面会更稳。” |
| 缺少合作意义 | “把感谢落到后续合作上，会更自然。” |
| 风险升高 | “这轮先把安全边界拉回来。” |

#### 6.3.5 输入区

输入区是核心操作区。

必须包含：

- 标签：你的回应
- 多行输入框
- 示例 placeholder
- 字数计数
- 发送回应按钮

禁止：

- A/B/C/D 选择项
- 多个并列主按钮
- 复杂格式工具栏

输入区交互：

- 输入少于 2 字时，发送按钮置灰。
- 输入超过限制时，不允许继续输入或提示限制。
- 提交后按钮显示“评估中”。
- 提交后进入反馈页。

### 6.4 AI 反馈页

目标：

- 让用户看见“我刚才那句话造成了什么效果”。

必须包含：

- 用户刚才输入的话
- NPC 动态回应
- NPC 情绪标签
- 状态变化标签
- 下一轮引导目标
- 即时点评
- 当前状态小结
- 继续按钮

反馈页结构：

```text
标题：现场反应
酒桌场景对话回放
├─ 林一头顶字幕：用户输入
└─ NPC 头顶字幕：动态回应
状态变化
下一轮引导目标
即时点评
当前状态文字与状态条
继续按钮
```

展示要求：

- 反馈页仍然以酒桌场景为主，不把对话主体验做成聊天列表。
- 用户刚才输入的话显示在林一头顶。
- NPC 动态回应显示在对应 NPC 头顶。
- 李总回应林一时，需要看向林一。
- 即时点评和状态变化可以使用侧边或下方面板，但不要抢走对话焦点。
- 反馈页需要明确告诉用户下一轮练习方向，例如“下一轮目标：把拒酒理由和合作心意连起来”。
- 引导目标不能是可点击选项，也不能自动填入输入框。

动效：

- 林一头顶字幕淡入。
- NPC 头顶字幕逐条出现，间隔 120-180ms。
- 状态变化标签轻微弹入。
- 状态条平滑增长或下降。

### 6.5 结局页

目标：

- 给用户一个明确收束。
- 让用户带走可复用知识。

必须包含：

- 结局名称
- 结局剧情
- 最终状态
- 本局复盘
- 知识卡
- 推荐话术
- 再玩一次按钮

通关结局额外要求：

- 「分寸拿捏」通关页必须展示李总最终认可话术。
- 该话术需要显示为李总头顶字幕或结局剧情重点台词。
- 通关页需要说明“客户已经接受边界，并认可合作态度”。
- 如果李总没有输出合格认可话术，即使状态较好，也不能展示为通关结局。

结局页语气：

- 不羞辱用户
- 不制造失败惩罚感
- 强调“下次可以怎么说”

知识卡展示：

```text
饭局拒酒四步法
承接好意 → 明确边界 → 给出替代 → 抬高意义
```

## 7. 组件规范

### 7.1 按钮

主按钮：

```text
高度：44px 以上
圆角：8px
背景：#23645B
文字：白色
字重：700
```

状态：

| 状态 | 表现 |
| --- | --- |
| 默认 | 主品牌色 |
| hover | 轻微上浮，颜色加深 |
| disabled | 透明度降低，不可点击 |
| loading | 文案变为“评估中” |

### 7.2 输入框

```text
最小高度：118px
圆角：8px
内边距：14px
边框：浅灰绿色
聚焦态：主色外发光
```

输入框需要给用户安全感，不能像考试答题框。

### 7.3 标签

用于学习点、状态变化、情绪。

```text
圆角：8px
内边距：8px 10px
字号：13-14px
背景：浅色
文字：对应语义色
```

### 7.4 面板

用于当前局势、状态反馈、复盘、知识卡。

```text
背景：#FFFAF0
边框：轻线
圆角：8px
阴影：轻量
```

面板不要过多嵌套。

### 7.5 头顶对话字幕 / 气泡

用途：

- 李总发起劝酒
- NPC 动态回应
- 用户输入后的林一发言

规范：

- 背景使用纸张白或轻提示色。
- 气泡或字幕必须锚定在说话角色头顶。
- 重要 NPC 台词可加粗。
- 角色名和情绪标签分开显示。
- 气泡出现时有轻微弹入动画。
- 气泡尾巴或位置关系要指向说话角色。
- 长文本自动换行，单个气泡建议不超过 3 行。
- 同一时间避免多个大气泡同时展开。
- 林一的气泡使用“你：林一”标识，颜色可略偏青绿，便于和 NPC 区分。

## 8. 动效规范

### 8.1 动效原则

动效用于帮助理解状态变化，不用于炫技。

原则：

- 轻
- 短
- 可预期
- 不干扰输入
- 不造成晕动

### 8.2 基础时长

| 动效 | 时长 |
| --- | --- |
| 页面淡入 | 180-300ms |
| 按钮 hover | 120-160ms |
| 气泡出现 | 180-240ms |
| 李总开场动态台词 | 1.8-2.8 秒 |
| 角色表情切换 | 160-260ms |
| 头顶字幕切换 | 180-260ms |
| NPC 回复逐条出现 | 每条间隔 120-180ms |
| 状态条变化 | 240-400ms |
| 反馈标签弹入 | 160-220ms |
| 当前局势滚动字幕 | 4-6 秒播放一遍 |
| 信息抽屉展开/收起 | 160-240ms |

### 8.3 场景动效

酒桌主界面建议包含：

- 包间暖光轻微呼吸
- 人物头像轻微上下浮动
- 人物表情根据情绪状态切换，例如施压、缓和、认可、尴尬
- 李总酒杯轻微抬起
- 李总开场台词以头顶动态字幕或动态气泡播放
- 用户发送后，林一头顶展示玩家输入内容
- NPC 说话时，对应角色头顶展示台词
- 李总说话时视线和身体朝向林一
- 茶杯轻微高亮，暗示“以茶代酒”
- 对话气泡出现时轻微弹入
- 风险升高时状态条轻微闪一次红色
- 当前局势顶部滚动字幕首轮播放一次，结束后收起到左下角
- 左下角信息抽屉 hover 时轻量展开

### 8.4 禁止动效

- 大幅晃动
- 快速闪烁
- 长时间循环抢注意力
- 夸张变形或惊吓式表情切换
- 按钮或输入框位置漂移
- 影响用户阅读的背景动画

### 8.5 减少动态模式

需要兼容 `prefers-reduced-motion`。

当用户系统开启减少动态时：

- 关闭浮动和呼吸动画。
- 保留必要的淡入。
- 状态条仍可变化，但不使用弹跳。

## 9. 易操作规范

### 9.1 单一主操作

每个页面只保留一个最重要的主操作：

| 页面 | 主操作 |
| --- | --- |
| 首页 | 开始游戏 |
| 场景介绍 | 进入酒桌 |
| 游戏主界面 | 发送回应 |
| 反馈页 | 继续 |
| 结局页 | 再玩一次 |

### 9.2 点击目标

- 按钮高度不低于 44px。
- 主按钮宽度需要与页面区域匹配，不能只包住文字。
- 点击区域不能只包住文字。

### 9.3 输入辅助

为了降低用户不知道怎么说的压力，输入框 placeholder 可以提供示例：

```text
请输入你要对李总说的话……
例如：李总，我今晚开车，酒真不能碰，但这杯茶我认真敬您，也感谢您对项目的支持。
```

注意：

- placeholder 只能提示，不要变成选项。
- 不要在界面上给出多个可点击标准答案。

### 9.4 错误和空输入

空输入：

```text
按钮置灰，不弹错误。
```

过短输入：

```text
按钮置灰，提示用户至少输入一句完整回应。
```

系统评估失败：

```text
显示轻提示：“刚才没听清，可以换个说法再试一次。”
保留用户输入，不清空。
```

## 10. 桌面 Web 布局规范

### 10.1 桌面端

推荐断点：

```text
推荐画布：1440px 宽桌面网页
最小支持宽度：1180px
```

桌面端要求：

- 酒桌场景横向铺开。
- 当前局势先以顶部滚动字幕播放，播放完成后进入左下角信息抽屉。
- 状态反馈和小贴士默认收纳在左下角信息抽屉中。
- 输入区在底部完整展示。
- 首屏尽量展示到输入区。

### 10.2 桌面注意事项

- 页面以电脑浏览器为唯一目标。
- 角色围绕酒桌横向排布，头像、字幕和输入区不得重叠。
- 左下角信息抽屉固定为 hover / focus 展开。
- 底部输入区需要始终可见或在首屏内清晰可达。
- 浏览器窗口缩窄到最小支持宽度时，需要检查：
- 酒桌区域是否无重叠。
- 头顶对话气泡是否完整显示。
- 剧情文本是否换行正常。
- 输入框是否可用。
- 信息抽屉展开后是否不遮挡输入框。

## 11. 可访问性规范

### 11.1 对比度

- 正文与背景对比度应满足 WCAG AA。
- 风险红不能只靠颜色表达，需要配合文字“风险”。
- 状态条旁必须显示文字状态，不显示具体数值；风险红不能只靠颜色表达，需要配合“风险较低 / 风险较高”等文字。

### 11.2 键盘操作

用户必须可以通过键盘完成：

```text
Tab 到按钮
Tab 到输入框
Tab 到左下角信息抽屉入口
展开当前局势 / 状态反馈 / 小贴士
输入回应
提交
继续
再玩一次
```

### 11.3 语义结构

- 页面主区域使用 `main`。
- 面板使用 `section`。
- 角色卡片使用可读的 aria-label。
- 输入框必须有 label。
- 按钮文案要明确，不使用只有图标的按钮。

## 12. 文案规范

### 12.1 语气

整体语气：

- 轻松
- 友好
- 像教练
- 不说教
- 不羞辱

### 12.2 反馈文案原则

反馈需要解释“为什么”，而不是只说“好”或“不好”。

示例：

```text
你给出了明确理由，也用以茶代酒保住了现场礼数。表达里有边界、有替代、有尊重。
```

避免：

```text
错误。
你不会做人。
这个回答很差。
```

### 12.3 风险文案

风险提醒要明确但不恐吓。

推荐：

```text
你短期给了面子，却放松了安全底线。开车饮酒是高风险选择。
```

避免：

```text
你完了。
你把事情搞砸了。
```

## 13. 场景插画规范

### 13.1 人物头像

MVP 可使用 CSS / SVG 生成头像，不依赖真实图片。

头像需要满足：

- 五个角色一眼可区分。
- 表情能表达基本情绪。
- 头像风格统一，可使用头肩像、半身像或上半身插画。
- 不使用圆形头像框、圆圈边框或卡片式头像框。
- 人物边缘可用自然描边或轻阴影区分背景。
- 不使用真实名人脸。
- 不使用过度夸张或冒犯性的面部特征。

### 13.2 酒桌物件

必须出现：

- 圆桌
- 菜品
- 茶杯
- 酒杯

表达原则：

- 茶杯和酒杯要视觉区分。
- 酒杯不能成为视觉中心，避免强化饮酒。
- 茶杯可在成熟回应路径中轻微高亮。

### 13.3 包间背景

背景建议：

- 温暖灯光
- 柔和墙面
- 轻微空间感

避免：

- 昏暗压抑
- 真实烟酒饭局质感
- 高饱和红金奢华感

## 14. 开发实现建议

### 14.1 CSS 变量

建议建立基础设计 token：

```css
:root {
  --color-brand: #23645b;
  --color-accent: #187781;
  --color-bg: #f6f2e8;
  --color-panel: #fffaf0;
  --color-text: #1d2525;
  --color-muted: #65716e;
  --color-good: #1f8a5b;
  --color-warm: #c4812d;
  --color-risk: #c64c3c;
  --radius-ui: 8px;
  --shadow-soft: 0 18px 55px rgba(43, 37, 30, 0.14);
}
```

### 14.2 动画实现

MVP 优先使用 CSS animation / transition。

推荐：

- `transform`
- `opacity`
- `width`

避免频繁动画：

- `top`
- `left`
- `height`
- 大面积 filter

### 14.3 后续可升级

后续如需更强动画感，可加入：

- Lottie 角色表情
- SVG 分层动画
- Canvas 轻量场景动画
- 角色表情状态机

MVP 阶段不强制引入复杂动画库。

## 15. UI 验收标准

### 15.1 风格验收

- 整体看起来像轻松的动画互动游戏。
- 不像考试系统、管理后台或严肃培训页面。
- 酒桌场景有动画插画感。
- 人物头像清晰、有角色差异。
- 页面氛围轻松，但职场饭局场景明确。

### 15.2 操作验收

- 用户从首页到主界面不需要学习复杂操作。
- 主界面只有自由输入，不出现预设选项。
- 主界面支持 3-6 轮多轮对话，而不是固定两轮。
- 每轮都有清晰的引导目标，但不展示标准答案按钮。
- 发送按钮清楚可见。
- 空输入不能提交。
- 反馈页能清楚看到 NPC 回应、状态变化和点评。
- 结局页能清楚看到复盘、推荐话术和客户最终认可话术。

### 15.3 桌面 Web 验收

- 1440px 桌面宽度下，酒桌场景、局势、状态、输入区布局完整。
- 1180px 最小桌面宽度下，头像不重叠，文本不横向溢出。
- 电脑浏览器中可以顺畅完成输入和继续操作。

### 15.4 动效验收

- 页面加载、按钮 hover、气泡出现、状态条变化有轻量动效。
- 进入游戏主界面后，李总开场台词有动态字幕或动态气泡。
- 李总开场说话和回应林一时，需要看向林一。
- 用户输入后，林一头顶会展示玩家输入内容。
- NPC 对话会显示在对应角色头顶，而不是主聊天列表中。
- 角色表情能随情绪状态变化，例如施压、缓和、认可、尴尬。
- 动效不会影响阅读。
- 风险变化有提示，但不恐吓。
- 开启减少动态后页面仍可正常使用。

## 16. 设计结论

《会来事儿》的 UI 不是要复刻真实酒桌，而是要把复杂人情场景变成一个轻松、可观察、可练习的动画互动空间。

首个场景「以茶代酒」的 UI 核心应是：

```text
动画化酒桌
+ 可识别角色
+ 清楚局势
+ 单一输入
+ 轻量反馈
+ 温和复盘
```

最终用户感受到的不是“我被考了”，而是：

```text
我刚刚在一个安全的动画场景里
练习了一句现实中真的用得上的话
```
