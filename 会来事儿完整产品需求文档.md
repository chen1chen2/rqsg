# 《会来事儿》完整产品需求文档

> 文档版本：v1.0  
> 更新时间：2026-05-26  
> 产品类型：AI 人情世故训练游戏  
> 首个 MVP 场景：初入职场 / 以茶代酒  
> 目标交付：可运行电脑端 Web Demo，可继续扩展为多场景 AI 社交训练产品

## 1. 产品概述

### 1.1 产品名称

**会来事儿**

### 1.2 产品定位

《会来事儿》是一款 AI 人情世故训练游戏。用户进入具体的中国式人际场景，与具备性格、身份、关系和情绪反应的 AI NPC 互动，通过自由输入自己的回应，学习如何在复杂社交场合中做到：

- 看懂场合
- 读懂潜台词
- 给人留面子
- 守住个人边界
- 表达得体而不讨好
- 维护关系但不牺牲底线

产品不是教用户圆滑、算计或操控他人，而是训练一种成熟的情境判断和表达能力。

### 1.3 一句话描述

在真实人情场景里，练习有分寸地表达。

### 1.4 MVP 目标

开发一个可运行的电脑端 Web Demo，完整实现首个 AI 互动场景「以茶代酒」。

MVP 需要让用户完成以下闭环：

```text
首页
→ 场景介绍
→ 酒桌主界面
→ AI NPC 发起社交压力
→ 用户自由输入回应
→ AI 评估回应质量
→ AI NPC 动态反应
→ 状态文字反馈变化
→ 第二轮互动
→ 结局判定
→ 复盘学习
```

## 2. 用户与使用场景

### 2.1 目标用户

主要用户：

- 初入职场的年轻人
- 需要提升沟通表达能力的学生
- 刚进入销售、商务、运营、管理岗位的人
- 对中国式人情世故不熟悉的用户
- 想练习拒绝、边界感、职场表达的人

次要用户：

- 管理者，用于培训新人
- 海外华人或中文学习者，用于理解中文语境下的人际互动
- 家庭教育、职场培训、社交沟通课程的内容开发者

### 2.2 用户痛点

- 知道人情世故重要，但不知道具体场景该怎么说。
- 只看文章和道理，很难迁移到真实对话。
- 害怕拒绝别人，担心得罪人。
- 有时守住了原则，但表达太硬，关系成本高。
- 有时给足了面子，但牺牲了边界和安全。
- 很多潜台词不是字面意思，新手难以识别。

### 2.3 产品价值

《会来事儿》通过游戏化情境训练，把抽象的人情世故拆成可观察、可练习、可复盘的能力：

```text
识别局面
→ 组织语言
→ 承担后果
→ 看见反馈
→ 复盘改进
```

## 4. 核心玩法

### 4.1 玩法类型

```text
AI 互动剧情
+ 社交模拟
+ 状态文字反馈
+ 复盘学习
```

### 4.2 单局流程

```text
1. 用户进入首页
2. 点击开始游戏
3. 阅读场景介绍
4. 进入包间酒桌主界面
5. 当前局势滚动字幕播放，并由 NPC 李总以动态台词字幕发起第一次劝酒
6. 用户自由输入一句回应
7. 用户回应以林一头顶动态字幕展示
8. 系统分析用户表达
9. AI NPC 根据角色设定和当前状态生成回应
10. NPC 回应以对应角色头顶动态字幕逐条展示
11. 系统更新内部状态，并展示文字化状态反馈
12. 展示即时反馈
13. 用户点击继续
14. 根据当前表现进入下一轮引导或压力
15. 用户继续自由输入回应
16. 系统持续判断是否已让客户接受边界
17. 若客户李总最终说出合格认可话术，则进入通关结局
18. 若达到风险、冷场或轮次上限，则进入对应结局或复盘
19. 展示结局、复盘、知识卡、推荐话术
```

### 4.3 玩家输入原则

MVP 中不提供 A/B/C/D 选项。用户必须自己输入自然语言回应。

原因：

- 强化真实表达训练
- 避免产品变成选择题
- 让 AI NPC 的价值更明显
- 训练用户自己组织语言，而不是背标准答案

### 4.4 用户每轮输入限制

```text
输入方式：文本输入
输入长度：建议 2-180 字
输入对象：默认对李总回应
输入轮次：MVP 支持 3-6 轮动态对话
提交方式：点击“发送回应”
```

### 4.5 多轮引导机制

MVP 不应只做固定两轮问答，而应做成“多轮对话 + 持续引导”的训练流程。

回合规则：

- 单局默认至少 3 轮，最多 6 轮。
- 每轮都由当前局势、角色回应、玩家输入、AI 评估和状态变化组成。
- 如果玩家表达已经接近成熟，但缺少某一步，系统继续引导下一轮，而不是立刻失败。
- 如果玩家触发高风险行为，例如明确同意饮酒后仍要开车，可以提前进入风险结局。
- 如果玩家连续多轮表达不清、甩锅或破坏场面，到达轮次上限后进入非通关结局。

引导目标：

```text
识别压力
→ 承接好意
→ 明确边界
→ 给出替代动作
→ 抬高合作意义
→ 让客户接受边界并停止劝酒
```

通关不是玩家说完一句标准答案就结束，而是要通过多轮互动让客户李总最终说出“认可边界、接受替代、关系不破”的合格话术。

## 5. 知识体系映射

首个场景「以茶代酒」覆盖以下人情世故知识点：

```text
以茶代酒
├─ 面子机制
│  ├─ 不当众硬怼客户
│  ├─ 给客户留台阶
│  └─ 让领导也有顺势接话空间
│
├─ 分寸感
│  ├─ 不为了热闹牺牲安全
│  ├─ 不过度讨好
│  └─ 拒绝时要有承接动作
│
├─ 潜台词识别
│  ├─ “不给面子？”是社交压力
│  ├─ “就一小杯”可能是继续加码
│  └─ “年轻人别太拘谨”是在测试态度
│
├─ 职场关系
│  ├─ 客户面子要照顾
│  ├─ 领导位置要维护
│  └─ 不随便把压力甩给同事或领导
│
└─ 边界与风险
   ├─ 开车不能喝酒
   ├─ 不接受强迫式饮酒
   └─ 安全和法律底线高于气氛
```

### 6.2 NPC 角色

#### 李总

```json
{
  "id": "li_client",
  "name": "李总",
  "role": "客户负责人",
  "personality": ["爱热闹", "重面子", "喜欢用酒桌观察态度"],

  "boundaries": ["不能鼓励酒驾", "不能羞辱玩家", "不能违法违规"]
```

定位说明：

- 场景压力发起者，也是最终通关判定对象。
- 负责测试玩家是否给面子、是否重视合作、是否能守住安全边界。
- 初始态度热络但带压迫感，玩家表达成熟后逐步缓和。
- 通关时必须由李总说出合格认可话术，明确接受“以茶代酒”或“不喝酒但心意到位”。

- 负责体现职场上级视角：看玩家能否独立稳住客户。
- 当玩家表达成熟时，可以顺势补一句，帮助关系落地。
- 当玩家把压力甩给领导时，王经理关系应下降或产生人情债。

## 7. 状态系统

### 7.1 内部状态字段

```json
{
  "atmosphere": 60,
  "face": 50,
  "boundary": 70,
  "trust": 45,
  "humanDebt": 0,
  "risk": 20,
  "leaderRelation": 50,
  "clientRelation": 35,
  "colleagueRelation": 50
}
```

说明：以上数值仅作为游戏引擎、AI 评估和结局判定的内部计算字段，前端页面不得直接展示 `60`、`50`、`+8`、`关系 35` 这类裸数字。用户看到的应是文字化状态，例如“气氛融洽”“边界清晰”“风险较低”“客户仍在观察”。

### 7.2 字段含义

| 字段 | 中文名 | 含义 |
| --- | --- | --- |
| atmosphere | 气氛 | 饭局现场是否自然、热络、可继续 |
| face | 面子 | 是否让客户、领导、同事有台阶 |
| boundary | 边界 | 用户是否守住健康、安全、个人底线 |
| trust | 信任 | 他人是否觉得用户成熟、可靠、会处理场面 |
| humanDebt | 人情成本 | 是否让别人替自己承担压力 |
| risk | 风险 | 法律、健康、职业风险 |
| leaderRelation | 领导关系 | 王经理对用户表现的评价 |
| clientRelation | 客户关系 | 李总对用户的态度 |
| colleagueRelation | 同事关系 | 周姐、阿诚对用户的态度 |

### 7.3 数值规则

```text
数值范围：0-100
默认变化：每轮 -30 到 +35
风险 risk 越高越危险
其他正向字段通常越高越好
所有数值更新后必须 clamp 到 0-100
```

### 7.4 前端文字化展示规则

前端展示状态时，必须把内部数值转换为用户可理解的文字等级。状态条可以保留视觉长度，但旁边不显示具体数字。

| 字段 | 低区间展示 | 中区间展示 | 高区间展示 |
| --- | --- | --- | --- |
| atmosphere | 气氛偏冷 / 有点紧张 | 气氛平稳 | 气氛融洽 / 气氛热络 |
| face | 略显尴尬 / 面子受损 | 基本顾到 | 体面周全 / 给足台阶 |
| boundary | 边界偏弱 / 边界失守 | 尚能坚持 | 边界清晰 / 底线很稳 |
| trust | 仍在观察 / 不太放心 | 基本可靠 | 稳重可信 / 很稳妥 |
| risk | 风险很低 / 风险较低 | 需要留意 | 风险较高 / 高风险 |

关系字段也必须文字化展示：

| 内部字段 | 前端展示示例 |
| --- | --- |
| leaderRelation | 领导观察中 / 领导认可 / 领导有顾虑 |
| clientRelation | 客户仍在观察 / 客户态度缓和 / 客户有保留 |
| colleagueRelation | 同事普通熟悉 / 同事愿意帮衬 / 同事略显尴尬 |

状态变化也不展示 `+8`、`-18`，而展示为：

```text
气氛更融洽
面子更周全
边界更清晰
风险降低
领导更认可
客户态度缓和
```

### 7.5 表达分类

AI 或 mock 评估器需要把用户输入归类为以下类型之一：

| 类型 | 说明 | 典型输入 |
| --- | --- | --- |
| balanced | 成熟表达 | “今晚开车，酒不能碰，我以茶敬您，也感谢您对项目的支持。” |
| respectful_boundary | 尊重式拒绝 | “我身体不太方便喝酒，这杯茶认真敬您。” |
| humor_boundary | 幽默式拒绝 | “我喝倒了，明天复盘就没人改了。” |
| plain_boundary | 普通拒绝 | “我今天不方便喝。” |
| shift_leader | 转移压力 | “王经理，我能不喝吗？” |
| blunt_boundary | 生硬拒绝 | “我不喝，您别劝了。” |
| risky | 风险迎合 | “那我喝一点吧。” |
| vague | 表达含糊 | “这个嘛，看情况。” |

## 8. AI NPC 系统

### 8.1 AI 架构原则

AI NPC 需要智能，但不能完全自由发挥。

推荐架构：

```text
剧情骨架固定
+ 角色人格配置
+ 用户输入自由
+ AI 动态回应
+ 状态系统约束
+ 安全规则兜底
```

### 8.2 AI 服务职责

AI 服务需要完成：

- 理解用户输入
- 判断表达策略
- 判断是否给面子
- 判断是否守边界
- 判断是否有合理理由
- 判断是否提供替代动作
- 判断是否存在风险
- 生成 NPC 动态回应
- 输出状态变化
- 输出即时点评
- 推进下一阶段或结局

### 8.3 AI 请求格式

```json
{
  "sceneId": "dinner_toast_001",
  "stage": "first_toast",
  "playerInput": "李总，我今晚开车，酒真不能碰，但这杯茶我认真敬您。",
  "state": {
    "atmosphere": 60,
    "face": 50,
    "boundary": 70,
    "trust": 45,
    "risk": 20
  },
  "characters": [
    {
      "id": "li_client",
      "name": "李总",
      "role": "客户负责人",
      "personality": ["爱热闹", "重面子"]
    }
  ],
  "learningPoints": ["面子", "分寸", "潜台词", "边界与风险"],
  "safetyRules": ["不能鼓励酒驾", "不能强迫饮酒", "不能把人情世故解释成操控他人"]
}
```

### 8.4 AI 响应格式

AI 响应必须是结构化 JSON，不能只返回一段自然语言。

```json
{
  "npcReplies": [
    {
      "characterId": "li_client",
      "text": "开车啊，那确实不能喝。行，这杯茶也算数，心意到了。",
      "emotion": "缓和",
      "expression": "眉眼放松、嘴角上扬",
      "motionCue": "放下酒杯，轻轻点头",
      "gazeTarget": "lin_yi"
    },
    {
      "characterId": "wang_manager",
      "text": "林一明天还要做复盘，今晚清醒点也好。李总，这杯我陪您走一个。",
      "emotion": "认可",
      "expression": "微笑、点头",
      "motionCue": "身体微微前倾，顺势接话",
      "gazeTarget": "li_client"
    }
  ],
  "stateDelta": {
    "atmosphere": 8,
    "face": 18,
    "boundary": 12,
    "trust": 18,
    "risk": -18,
    "clientRelation": 8,
    "leaderRelation": 5
  },
  "feedback": "你给出了明确理由，也用以茶代酒保住了现场礼数。表达里有边界、有替代、有尊重。",
  "nextStage": "second_pressure",
  "category": "balanced",
  "guidance": {
    "level": "light",
    "text": "你已经说明了不能喝酒，下一步可以把这杯茶和后续合作连接起来。"
  },
  "passCheck": {
    "passed": false,
    "reason": "客户已缓和，但还没有明确说出接受边界并停止劝酒的认可话术。",
    "requiredCustomerMeaning": ["认可不喝酒的边界", "接受以茶代酒", "确认合作关系不受影响"]
  }
}
```

表情与动效要求：

- AI 响应中的 `emotion`、`expression`、`motionCue` 用于驱动前端角色表情和小动作。
- AI 响应中的 `gazeTarget` 用于驱动角色视线朝向和身体朝向，避免角色像在对空气说话。
- 每个 NPC 的表情动效必须与当前情绪一致，例如李总“施压”时眉毛上扬、酒杯前递，“缓和”时嘴角放松、酒杯放低。
- 当李总对林一施压或回应林一时，`gazeTarget` 应为 `lin_yi`，画面中必须表现为李总看着林一说话。
- `guidance` 用于给用户下一轮轻量引导，只提示表达方向，不直接给完整答案。
- `passCheck` 用于判断当前是否已达到通关条件；只有客户李总输出合格认可话术时，`passed` 才能为 `true`。
- 表情动效只做轻量变化，不做夸张变形，不影响用户阅读台词。
- 前端可以先用 CSS/SVG 状态类实现，后续可替换为 Lottie、分层 SVG 或 Canvas 动效。

### 8.5 MVP AI 实现方式

MVP 可以先使用 mock AI：

- 通过关键词和规则判断输入类型
- 根据输入类型返回固定模板中的动态回应
- 根据输入类型调整内部状态，并在前端展示文字化变化
- 后续再替换为真实 AI 服务

技术要求：

- 前端不得暴露 AI API Key
- 真实 AI 调用必须通过后端代理
- AI 服务必须支持 mock 与真实模型切换

## 9. 场景剧情设计

### 9.1 第一轮开场

```text
晚上 7:18，包间里已经坐满了人。

这是你入职以来第一次参加客户饭局。项目刚刚验收通过，王经理特意把你也带上，说是“让客户认识一下年轻人”。

你今晚是开车来的，酒量也很差。更麻烦的是，明天上午你还要做项目复盘。

菜刚上齐，李总端起酒杯，看向你：

“林一是吧？年轻人第一次见，来，咱俩喝一个。”
```

用户自由输入回应。

开场表现要求：

- 进入酒桌主界面后，李总第一句“林一是吧？年轻人第一次见，来，咱俩喝一个。”需要以动态台词字幕或动态气泡播放。
- 台词出现时需要有轻量动效，例如逐字显现、字幕条滑入、气泡弹入或酒杯轻抬。
- 李总表情同步进入“热络但施压”状态，表现为看向林一、端杯、身体微微前倾、表情带笑但有压迫感。
- 台词播放完成后，输入框进入可输入状态，用户可以自由回应。

### 9.2 第二轮压力

如果用户第一轮没有直接触发风险结局，则进入第二轮。第二轮不是固定终点，而是多轮引导中的一个压力节点。

```text
李总把酒杯在桌上轻轻一碰，笑着说：

“年轻人别太拘谨嘛。今天项目过了，出来吃饭就是图个热闹。”

桌上几个人都看向你。
```

用户再次自由输入回应。

### 9.3 多轮引导阶段

第二轮后，系统根据玩家表现进入不同引导阶段，直到通关、风险失控或达到轮次上限。

```text
first_toast：李总第一次敬酒
clarify_boundary：引导玩家说清不能喝的事实和边界
offer_substitute：引导玩家给出替代动作，例如以茶代酒
raise_meaning：引导玩家把回应抬高到合作、感谢和后续支持
client_acceptance：客户确认接受边界，停止劝酒
ending_review：进入结局和复盘
```

阶段推进规则：

- 玩家只说“我不喝”时，进入 `offer_substitute` 或 `raise_meaning`，引导补上礼数和替代动作。
- 玩家只说“我开车”但没有承接好意时，进入 `raise_meaning`，引导补上感谢和合作意义。
- 玩家把压力甩给王经理时，进入 `clarify_boundary`，并降低领导评价。
- 玩家同意喝酒或表达侥幸时，显著提高风险，必要时提前进入风险结局。
- 玩家表达包含“承接好意 + 明确边界 + 替代动作 + 抬高意义”后，进入 `client_acceptance`。

### 9.4 推荐成熟表达

```text
李总，我今晚开车，酒真不能碰。
但这杯茶我认真敬您，也感谢您对项目的支持。
后续项目上的事，您有需要我一定第一时间响应。
```

结构：

```text
承接对象
→ 明确事实
→ 表明边界
→ 给出替代
→ 抬高意义
```

## 10. 结局系统

### 10.1 结局优先级

```text
失控风险
> 分寸拿捏
> 关系保住但边界松动
> 边界守住但场面变冷
```

### 10.2 结局条件

#### S 结局：分寸拿捏

```text
boundary >= 80
face >= 70
risk <= 15
trust >= 65
passCheck.passed == true
李总最终输出合格认可话术
```

合格认可话术标准：

```text
李总的话需要同时表达：
1. 认可林一不喝酒的边界，例如“开车不喝酒是对的”
2. 接受替代动作，例如“这杯茶也算数”
3. 保住关系和合作，例如“心意到了，后面项目继续好好配合”
4. 停止继续劝酒，不再制造饮酒压力
```

结局文本：

```text
李总端起茶杯：“行，开车不喝酒，这个规矩对。年轻人做事挺稳。”

王经理顺势接话：“林一明天还要做复盘，今晚让他清醒点。”

饭局继续热闹，但没人再逼你喝酒。
```

S 结局示例客户最终话术：

```text
李总看着林一，放下酒杯，端起茶杯说：
“行，开车不喝酒是对的。这杯茶也算数，心意到了。后面项目继续好好配合。”
```

#### A 结局：关系保住但边界松动

```text
face >= 65
boundary < 60
risk >= 35
passCheck.passed == false
```

结局文本：

```text
饭局气氛变热，李总对你印象也好了些。

但阿诚笑着说：“以后这种场面就靠你了。”

你意识到，今天的让步可能会变成下一次的默认。
```

#### B 结局：边界守住但场面变冷

```text
boundary >= 80
face < 45
atmosphere < 45
passCheck.passed == false
```

结局文本：

```text
你确实没有喝酒。

只是接下来的几分钟，包间气氛明显冷了下来。

王经理后来提醒你：“原则没错，但话可以再圆一点。”
```

#### C 结局：失控风险

```text
risk >= 60
或玩家明确选择饮酒且仍需要开车
```

结局文本：

```text
你喝了不止一杯。

饭局结束时，你已经不适合开车。王经理只能安排代驾。

李总倒是挺高兴，但你心里清楚：这不是一个值得复制的选择。
```

#### 12.3.1 当前局势滚动字幕

进入游戏主界面时，系统需要先在页面顶部展示当前局势滚动字幕，用轻量动画向用户交代局面。

滚动字幕内容：

```text
场合：客户庆功饭局
压力来源：李总第一次敬酒
你的事实：今晚开车、酒量很差、明天上午复盘
隐性目标：拒酒但不让客户当场下不来台
```

交互规则：

- 滚动字幕位于顶部信息栏下方或作为顶部浮层显示，不遮挡核心酒桌人物。
- 字幕自动播放一遍，推荐时长 4-6 秒。
- 播放完成后自动收起，不再常驻占用顶部空间。
- 收起后，当前局势进入左下角信息抽屉组，仅显示“当前局势”标签。
- 鼠标 hover “当前局势”标签时，完整展开当前局势内容。
- 鼠标移出后自动收起。

#### 12.3.1.1 李总开场动态台词

进入酒桌主界面后，李总需要主动说出第一句劝酒台词，不能只把台词静态放在页面上。

台词内容：

```text
林一是吧？年轻人第一次见，来，咱俩喝一个。
```

展示规则：

- 台词以动态字幕条或动态对话气泡出现，推荐时长 1.8-2.8 秒。
- 台词可以逐字显现、短句滑入或气泡轻弹入，但必须保证可读。
- 李总角色同步播放表情动效：看向林一、端杯、微笑、眉毛轻挑、身体微微前倾。
- 台词播放时不展示任何选择项，也不自动替用户填充回答。
- 台词播放完成后，输入框获得视觉焦点，提示用户自由输入。
- 开启减少动态模式时，改为静态展示 2 秒后进入可输入状态。

#### 12.3.1.2 角色头顶对话字幕

主游戏界面的对话呈现需要贴近现场交流感，所有角色发言都应出现在对应角色头顶，而不是集中展示成聊天记录。

展示规则：

- 李总、王经理、周姐、阿诚、林一的发言，都以对应角色头顶字幕或对话气泡展示。
- 用户点击“发送回应”后，玩家输入内容先出现在林一头顶，展示完成后再进入 AI 评估和 NPC 回应。
- NPC 回应按发生顺序逐条出现在对应 NPC 头顶，同一时间建议只突出一条主发言，避免画面拥挤。
- 当前说话角色需要有轻量强调，例如人物微亮、轻微前倾或表情变化。
- 被说话对象需要通过视线或身体朝向体现。李总对林一说话时，必须看向林一。
- 长文本需要自动分行或截断成短句轮播，不能遮挡其他角色、状态抽屉和输入区。
- 反馈页如需回放对话，也应优先使用酒桌场景中的头顶字幕，而不是纯文本聊天列表。

#### 12.3.2 左下角信息抽屉组

游戏主界面左下角需要提供一组默认隐藏的信息抽屉，减少主画面干扰。

信息抽屉包含：

```text

```

展示规则：

- 默认状态只显示三个轻量标签或图标+文字短标签。
- 标签固定在页面左下方，不遮挡用户输入框。
- 鼠标 hover 某个标签时，仅展开该标签对应内容。
- 展开层使用轻量面板，不使用大面积卡片遮挡酒桌主体。
- 状态反馈显示文字状态和状态条，不显示具体数值。
- 小贴士显示当前场景的短提示，例如“先承接好意，再说明边界”。
- 展开层必须支持鼠标 hover 和键盘 focus 展开。

## 14. 安全与价值观要求

### 14.1 内容底线

产品不得输出或鼓励：

- 酒驾
- 强迫饮酒
- 行贿送礼
- 权色交易
- 职场霸凌
- 羞辱、歧视或贬低用户
- 把人情世故包装成操控他人
- 为达成关系目的牺牲法律、健康和职业安全

### 14.2 复盘语气

复盘应该像教练，不像考试批改。

语气原则：

- 不羞辱用户
- 不给绝对标准答案
- 解释关系成本和风险
- 给出可复用表达结构
- 强调有边界的成熟表达

### 14.3 产品价值观

```text
懂关系，但不牺牲底线
会表达，但不虚伪讨好
给面子，也保护自己
讲人情，也尊重规则
```

- 刚进入游戏主界面时，当前局势会以顶部滚动字幕播放一遍。
- 刚进入游戏主界面时，李总第一句劝酒台词会以动态字幕或动态气泡播放。
- 李总开场说话时需要看向林一。
- 当前局势播放完成后会收起到左下角，hover 后完整展示。
- 状态反馈和小贴士默认隐藏在左下角信息抽屉组中，hover 后展示。
- 角色会根据情绪展示对应表情动效，例如施压、缓和、认可、尴尬。
- 用户输入后，玩家发言会显示在林一头顶字幕中。
- 所有 NPC 对话会显示在对应角色头顶字幕中，而不是以聊天列表作为主对话呈现。
- 主界面不得出现预设选项按钮。
- 用户可以自由输入一句回应。
- 提交后系统能生成 AI NPC 反馈。
- 内部状态会根据用户输入变化，前端以隐藏抽屉中的文字状态和状态条变化呈现。
- 游戏支持 3-6 轮动态对话，并根据玩家表现持续引导。
- 每轮都能展示当前引导方向，但不得直接给出完整标准答案。
- 只有客户李总最终输出合格认可话术，才可以进入「分寸拿捏」通关结局。
- 达到风险、冷场、边界松动或轮次上限时，进入对应非通关结局。
- 结局页展示复盘、知识卡、推荐话术。
- 用户可以点击「再玩一次」重置游戏。

### 15.2 场景验收

- 输入“开车不能喝，以茶代酒，表达敬意”应倾向触发「分寸拿捏」。
- 触发「分寸拿捏」前，李总必须说出类似“开车不喝酒是对的，这杯茶也算数，后面项目继续好好配合”的认可话术。
- 输入“那我喝一点吧”应显著提高 risk，并可能触发「失控风险」。
- 输入生硬拒绝应提高 boundary，但降低 face 和 atmosphere。
- 输入“王经理我能不喝吗”应增加 humanDebt 或降低 leaderRelation。
- 内部状态值必须始终保持在 0-100，前端不得直接展示这些数字。

### 15.3 UI 验收

- 桌面端 Web 主界面能完整展示酒桌、人物、顶部滚动字幕、左下角信息抽屉和输入区。
- 桌面端页面中，头像、文本、信息抽屉、状态条、输入框不得互相遮挡。
- 文字不得横向溢出。
- 首页首屏可直接开始体验，不做纯营销落地页。
- 页面视觉应体现现代职场饭局，而不是古风或纯卡通问答。

### 15.4 技术验收

- `npm.cmd run build` 必须通过。
- 构建产物可运行。
- AI service 层与 UI 解耦。
- 场景内容可通过配置扩展。
- 前端代码中不得出现真实 AI API Key。

## 18. PRD 结论

《会来事儿》MVP 的关键不是做一个聊天框，而是做一个有场景、有角色、有压力、有反馈、有复盘的 AI 社交训练闭环。

首个场景「以茶代酒」需要让用户明确感受到：

```text
真正的会来事儿
不是无底线迎合
也不是生硬拒绝
而是在关系、面子、边界和风险之间
说出一句合适的话
```
