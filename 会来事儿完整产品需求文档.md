# 《会来事儿》完整产品需求文档

> 文档版本：v1.0  
> 更新时间：2026-05-26  
> 产品类型：AI 人情世故训练游戏  
> 首个 MVP 场景：初入职场 / 以茶代酒  
> 目标交付：可运行 Web Demo，可继续扩展为多场景 AI 社交训练产品

## 1. 产品概述

### 1.1 产品名称

**会来事儿**

### 1.2 产品定位

《会来事儿》是一款 AI 人情世故训练游戏。用户进入具体的中国式人际场景，与具备性格、身份、关系和情绪反应的 AI NPC 互动，通过自由输入自己的回应，学习如何在复杂社交场合中做到：

- 看懂场合
- 读懂潜台词
- 给人留面子
- 守住个人边界
- 表达得体而不讨好
- 维护关系但不牺牲底线

产品不是教用户圆滑、算计或操控他人，而是训练一种成熟的情境判断和表达能力。

### 1.3 一句话描述

在真实人情场景里，练习有分寸地表达。

### 1.4 MVP 目标

开发一个可运行的 Web Demo，完整实现首个 AI 互动场景「以茶代酒」。

MVP 需要让用户完成以下闭环：

```text
首页
→ 场景介绍
→ 酒桌主界面
→ AI NPC 发起社交压力
→ 用户自由输入回应
→ AI 评估回应质量
→ AI NPC 动态反应
→ 状态文字反馈变化
→ 第二轮互动
→ 结局判定
→ 复盘学习
```

## 2. 用户与使用场景

### 2.1 目标用户

主要用户：

- 初入职场的年轻人
- 需要提升沟通表达能力的学生
- 刚进入销售、商务、运营、管理岗位的人
- 对中国式人情世故不熟悉的用户
- 想练习拒绝、边界感、职场表达的人

次要用户：

- 管理者，用于培训新人
- 海外华人或中文学习者，用于理解中文语境下的人际互动
- 家庭教育、职场培训、社交沟通课程的内容开发者

### 2.2 用户痛点

- 知道人情世故重要，但不知道具体场景该怎么说。
- 只看文章和道理，很难迁移到真实对话。
- 害怕拒绝别人，担心得罪人。
- 有时守住了原则，但表达太硬，关系成本高。
- 有时给足了面子，但牺牲了边界和安全。
- 很多潜台词不是字面意思，新手难以识别。

### 2.3 产品价值

《会来事儿》通过游戏化情境训练，把抽象的人情世故拆成可观察、可练习、可复盘的能力：

```text
识别局面
→ 组织语言
→ 承担后果
→ 看见反馈
→ 复盘改进
```

## 4. 核心玩法

### 4.1 玩法类型

```text
AI 互动剧情
+ 社交模拟
+ 状态文字反馈
+ 复盘学习
```

### 4.2 单局流程

```text
1. 用户进入首页
2. 点击开始游戏
3. 阅读场景介绍
4. 进入包间酒桌主界面
5. NPC 李总发起第一次劝酒
6. 用户自由输入一句回应
7. 系统分析用户表达
8. AI NPC 根据角色设定和当前状态生成回应
9. 系统更新内部状态，并展示文字化状态反馈
10. 展示即时反馈
11. 用户点击继续
12. 进入第二轮压力
13. 用户再次自由输入回应
14. 系统评估并判定结局
15. 展示结局、复盘、知识卡、推荐话术
```

### 4.3 玩家输入原则

MVP 中不提供 A/B/C/D 选项。用户必须自己输入自然语言回应。

原因：

- 强化真实表达训练
- 避免产品变成选择题
- 让 AI NPC 的价值更明显
- 训练用户自己组织语言，而不是背标准答案

### 4.4 用户每轮输入限制

```text
输入方式：文本输入
输入长度：建议 2-180 字
输入对象：默认对李总回应
输入轮次：MVP 至少 2 轮
提交方式：点击“发送回应”
```

## 5. 知识体系映射

首个场景「以茶代酒」覆盖以下人情世故知识点：

```text
以茶代酒
├─ 面子机制
│  ├─ 不当众硬怼客户
│  ├─ 给客户留台阶
│  └─ 让领导也有顺势接话空间
│
├─ 分寸感
│  ├─ 不为了热闹牺牲安全
│  ├─ 不过度讨好
│  └─ 拒绝时要有承接动作
│
├─ 潜台词识别
│  ├─ “不给面子？”是社交压力
│  ├─ “就一小杯”可能是继续加码
│  └─ “年轻人别太拘谨”是在测试态度
│
├─ 职场关系
│  ├─ 客户面子要照顾
│  ├─ 领导位置要维护
│  └─ 不随便把压力甩给同事或领导
│
└─ 边界与风险
   ├─ 开车不能喝酒
   ├─ 不接受强迫式饮酒
   └─ 安全和法律底线高于气氛
```

## 7. 状态系统

### 7.1 内部状态字段

```json
{
  "atmosphere": 60,
  "face": 50,
  "boundary": 70,
  "trust": 45,
  "humanDebt": 0,
  "risk": 20,
  "leaderRelation": 50,
  "clientRelation": 35,
  "colleagueRelation": 50
}
```

说明：以上数值仅作为游戏引擎、AI 评估和结局判定的内部计算字段，前端页面不得直接展示 `60`、`50`、`+8`、`关系 35` 这类裸数字。用户看到的应是文字化状态，例如“气氛融洽”“边界清晰”“风险较低”“客户仍在观察”。

### 7.2 字段含义

| 字段 | 中文名 | 含义 |
| --- | --- | --- |
| atmosphere | 气氛 | 饭局现场是否自然、热络、可继续 |
| face | 面子 | 是否让客户、领导、同事有台阶 |
| boundary | 边界 | 用户是否守住健康、安全、个人底线 |
| trust | 信任 | 他人是否觉得用户成熟、可靠、会处理场面 |
| humanDebt | 人情成本 | 是否让别人替自己承担压力 |
| risk | 风险 | 法律、健康、职业风险 |
| leaderRelation | 领导关系 | 王经理对用户表现的评价 |
| clientRelation | 客户关系 | 李总对用户的态度 |
| colleagueRelation | 同事关系 | 周姐、阿诚对用户的态度 |

### 7.3 数值规则

```text
数值范围：0-100
默认变化：每轮 -30 到 +35
风险 risk 越高越危险
其他正向字段通常越高越好
所有数值更新后必须 clamp 到 0-100
```

### 7.4 前端文字化展示规则

前端展示状态时，必须把内部数值转换为用户可理解的文字等级。状态条可以保留视觉长度，但旁边不显示具体数字。

| 字段 | 低区间展示 | 中区间展示 | 高区间展示 |
| --- | --- | --- | --- |
| atmosphere | 气氛偏冷 / 有点紧张 | 气氛平稳 | 气氛融洽 / 气氛热络 |
| face | 略显尴尬 / 面子受损 | 基本顾到 | 体面周全 / 给足台阶 |
| boundary | 边界偏弱 / 边界失守 | 尚能坚持 | 边界清晰 / 底线很稳 |
| trust | 仍在观察 / 不太放心 | 基本可靠 | 稳重可信 / 很稳妥 |
| risk | 风险很低 / 风险较低 | 需要留意 | 风险较高 / 高风险 |

关系字段也必须文字化展示：

| 内部字段 | 前端展示示例 |
| --- | --- |
| leaderRelation | 领导观察中 / 领导认可 / 领导有顾虑 |
| clientRelation | 客户仍在观察 / 客户态度缓和 / 客户有保留 |
| colleagueRelation | 同事普通熟悉 / 同事愿意帮衬 / 同事略显尴尬 |

状态变化也不展示 `+8`、`-18`，而展示为：

```text
气氛更融洽
面子更周全
边界更清晰
风险降低
领导更认可
客户态度缓和
```

### 7.5 表达分类

AI 或 mock 评估器需要把用户输入归类为以下类型之一：

| 类型 | 说明 | 典型输入 |
| --- | --- | --- |
| balanced | 成熟表达 | “今晚开车，酒不能碰，我以茶敬您，也感谢您对项目的支持。” |
| respectful_boundary | 尊重式拒绝 | “我身体不太方便喝酒，这杯茶认真敬您。” |
| humor_boundary | 幽默式拒绝 | “我喝倒了，明天复盘就没人改了。” |
| plain_boundary | 普通拒绝 | “我今天不方便喝。” |
| shift_leader | 转移压力 | “王经理，我能不喝吗？” |
| blunt_boundary | 生硬拒绝 | “我不喝，您别劝了。” |
| risky | 风险迎合 | “那我喝一点吧。” |
| vague | 表达含糊 | “这个嘛，看情况。” |

## 8. AI NPC 系统

### 8.1 AI 架构原则

AI NPC 需要智能，但不能完全自由发挥。

推荐架构：

```text
剧情骨架固定
+ 角色人格配置
+ 用户输入自由
+ AI 动态回应
+ 状态系统约束
+ 安全规则兜底
```

### 8.2 AI 服务职责

AI 服务需要完成：

- 理解用户输入
- 判断表达策略
- 判断是否给面子
- 判断是否守边界
- 判断是否有合理理由
- 判断是否提供替代动作
- 判断是否存在风险
- 生成 NPC 动态回应
- 输出状态变化
- 输出即时点评
- 推进下一阶段或结局

### 8.3 AI 请求格式

```json
{
  "sceneId": "dinner_toast_001",
  "stage": "first_toast",
  "playerInput": "李总，我今晚开车，酒真不能碰，但这杯茶我认真敬您。",
  "state": {
    "atmosphere": 60,
    "face": 50,
    "boundary": 70,
    "trust": 45,
    "risk": 20
  },
  "characters": [
    {
      "id": "li_client",
      "name": "李总",
      "role": "客户负责人",
      "personality": ["爱热闹", "重面子"]
    }
  ],
  "learningPoints": ["面子", "分寸", "潜台词", "边界与风险"],
  "safetyRules": ["不能鼓励酒驾", "不能强迫饮酒", "不能把人情世故解释成操控他人"]
}
```

### 8.4 AI 响应格式

AI 响应必须是结构化 JSON，不能只返回一段自然语言。

```json
{
  "npcReplies": [
    {
      "characterId": "li_client",
      "text": "开车啊，那确实不能喝。行，这杯茶也算数，心意到了。",
      "emotion": "缓和"
    },
    {
      "characterId": "wang_manager",
      "text": "林一明天还要做复盘，今晚清醒点也好。李总，这杯我陪您走一个。",
      "emotion": "认可"
    }
  ],
  "stateDelta": {
    "atmosphere": 8,
    "face": 18,
    "boundary": 12,
    "trust": 18,
    "risk": -18,
    "clientRelation": 8,
    "leaderRelation": 5
  },
  "feedback": "你给出了明确理由，也用以茶代酒保住了现场礼数。表达里有边界、有替代、有尊重。",
  "nextStage": "second_pressure",
  "category": "balanced"
}
```

### 8.5 MVP AI 实现方式

MVP 可以先使用 mock AI：

- 通过关键词和规则判断输入类型
- 根据输入类型返回固定模板中的动态回应
- 根据输入类型调整内部状态，并在前端展示文字化变化
- 后续再替换为真实 AI 服务

技术要求：

- 前端不得暴露 AI API Key
- 真实 AI 调用必须通过后端代理
- AI 服务必须支持 mock 与真实模型切换

## 9. 场景剧情设计

### 9.1 第一轮开场

```text
晚上 7:18，包间里已经坐满了人。

这是你入职以来第一次参加客户饭局。项目刚刚验收通过，王经理特意把你也带上，说是“让客户认识一下年轻人”。

你今晚是开车来的，酒量也很差。更麻烦的是，明天上午你还要做项目复盘。

菜刚上齐，李总端起酒杯，看向你：

“林一是吧？年轻人第一次见，来，咱俩喝一个。”
```

用户自由输入回应。

### 9.2 第二轮压力

如果用户第一轮没有直接触发风险结局，则进入第二轮。

```text
李总把酒杯在桌上轻轻一碰，笑着说：

“年轻人别太拘谨嘛。今天项目过了，出来吃饭就是图个热闹。”

桌上几个人都看向你。
```

用户再次自由输入回应。

### 9.3 推荐成熟表达

```text
李总，我今晚开车，酒真不能碰。
但这杯茶我认真敬您，也感谢您对项目的支持。
后续项目上的事，您有需要我一定第一时间响应。
```

结构：

```text
承接对象
→ 明确事实
→ 表明边界
→ 给出替代
→ 抬高意义
```

## 10. 结局系统

### 10.1 结局优先级

```text
失控风险
> 分寸拿捏
> 关系保住但边界松动
> 边界守住但场面变冷
```

### 10.2 结局条件

#### S 结局：分寸拿捏

```text
boundary >= 80
face >= 70
risk <= 15
trust >= 65
```

结局文本：

```text
李总端起茶杯：“行，开车不喝酒，这个规矩对。年轻人做事挺稳。”

王经理顺势接话：“林一明天还要做复盘，今晚让他清醒点。”

饭局继续热闹，但没人再逼你喝酒。
```

#### A 结局：关系保住但边界松动

```text
face >= 65
boundary < 60
risk >= 35
```

结局文本：

```text
饭局气氛变热，李总对你印象也好了些。

但阿诚笑着说：“以后这种场面就靠你了。”

你意识到，今天的让步可能会变成下一次的默认。
```

#### B 结局：边界守住但场面变冷

```text
boundary >= 80
face < 45
atmosphere < 45
```

结局文本：

```text
你确实没有喝酒。

只是接下来的几分钟，包间气氛明显冷了下来。

王经理后来提醒你：“原则没错，但话可以再圆一点。”
```

#### C 结局：失控风险

```text
risk >= 60
```

结局文本：

```text
你喝了不止一杯。

饭局结束时，你已经不适合开车。王经理只能安排代驾。

李总倒是挺高兴，但你心里清楚：这不是一个值得复制的选择。
```

## 11. 复盘系统

### 11.1 复盘页内容

结局页必须展示：

- 结局名称
- 结局剧情收束
- 最终状态文字与状态条
- 本局复盘
- 推荐话术
- 解锁知识卡
- 再玩一次按钮

2. 明确边界

## 12. 页面需求

### 12.1 首页

页面目标：

- 让用户知道产品是什么
- 快速开始体验

必须包含：

- 产品名：会来事儿
- 定位文案：在真实人情场景里，练习有分寸地表达。
- 开始游戏按钮
- 酒桌或人际场景视觉暗示

### 12.2 场景介绍页

页面目标：

- 告诉用户当前身份、场景、冲突和学习点

必须包含：

- 章节名：初入职场
- 场景名：以茶代酒
- 玩家身份说明
- 场景冲突说明
- 学习点标签
- 进入酒桌按钮

### 12.3 游戏主界面

主界面要求：

- 中央必须是包间酒桌场景，不是纯聊天框
- 必须展示圆形酒桌、菜品、茶杯、酒杯、包间暖光
- 必须展示人物头像
- 每个 NPC 显示姓名、身份、关系状态文字，不显示关系数字
- 刚进入游戏时，当前局势以顶部滚动字幕方式播放一遍
- 当前局势滚动字幕播放完成后，收起到页面左下方隐藏标签，鼠标 hover 后完整展示
- 状态反馈和小贴士默认隐藏在左下角信息抽屉组中，鼠标 hover 后展示完整内容
- 状态反馈展示文字状态和状态条，不显示裸数值
- 底部只保留自由输入框和发送按钮
- 不得出现 A/B/C/D 预设选项

桌面布局：

```text
顶部：产品名 / 章节 / 场景名 / 当前轮次
顶部浮层：当前局势滚动字幕，首轮进入时自动播放

中部：包间酒桌场景
├─ 王经理头像
├─ 李总头像
├─ 周姐头像
├─ 阿诚头像
├─ 玩家头像
└─ 酒桌、菜品、茶杯、酒杯

下部：
├─ 用户输入区
└─ 左下角信息抽屉组：当前局势 / 状态反馈 / 小贴士
```

移动端布局：

- 单列纵向布局
- 头像不得重叠
- 文本不得横向溢出
- 状态条、输入框、剧情文本不得互相遮挡
- 酒桌场景可纵向拉高，以保证角色可读
- 移动端没有 hover 时，左下角信息抽屉组改为点击展开/收起

#### 12.3.1 当前局势滚动字幕

进入游戏主界面时，系统需要先在页面顶部展示当前局势滚动字幕，用轻量动画向用户交代局面。

滚动字幕内容：

```text
场合：客户庆功饭局
压力来源：李总第一次敬酒
你的事实：今晚开车、酒量很差、明天上午复盘
隐性目标：拒酒但不让客户当场下不来台
```

交互规则：

- 滚动字幕位于顶部信息栏下方或作为顶部浮层显示，不遮挡核心酒桌人物。
- 字幕自动播放一遍，推荐时长 4-6 秒。
- 播放完成后自动收起，不再常驻占用顶部空间。
- 收起后，当前局势进入左下角信息抽屉组，仅显示“当前局势”标签。
- 鼠标 hover “当前局势”标签时，完整展开当前局势内容。
- 鼠标移出后自动收起。
- 移动端使用点击展开/收起替代 hover。

#### 12.3.2 左下角信息抽屉组

游戏主界面左下角需要提供一组默认隐藏的信息抽屉，减少主画面干扰。

信息抽屉包含：

```text
当前局势
状态反馈
小贴士
```

展示规则：

- 默认状态只显示三个轻量标签或图标+文字短标签。
- 标签固定在页面左下方，不遮挡用户输入框。
- 鼠标 hover 某个标签时，仅展开该标签对应内容。
- 展开层使用轻量面板，不使用大面积卡片遮挡酒桌主体。
- 状态反馈显示文字状态和状态条，不显示具体数值。
- 小贴士显示当前场景的短提示，例如“先承接好意，再说明边界”。
- 展开层必须支持键盘 focus 展开，移动端支持点击展开/收起。

### 12.4 反馈页

必须展示：

- 用户刚才输入的回应
- AI NPC 动态反应
- NPC 情绪标签
- 状态变化文字
- 即时点评
- 继续按钮

### 12.5 结局页

必须展示：

- 结局标题
- 结局剧情
- 最终状态文字
- 本局复盘
- 知识卡
- 推荐话术
- 再玩一次按钮

## 14. 安全与价值观要求

### 14.1 内容底线

产品不得输出或鼓励：

- 酒驾
- 强迫饮酒
- 行贿送礼
- 权色交易
- 职场霸凌
- 羞辱、歧视或贬低用户
- 把人情世故包装成操控他人
- 为达成关系目的牺牲法律、健康和职业安全

### 14.2 复盘语气

复盘应该像教练，不像考试批改。

语气原则：

- 不羞辱用户
- 不给绝对标准答案
- 解释关系成本和风险
- 给出可复用表达结构
- 强调有边界的成熟表达

### 14.3 产品价值观

```text
懂关系，但不牺牲底线
会表达，但不虚伪讨好
给面子，也保护自己
讲人情，也尊重规则
```

## 15. 验收标准

### 15.1 功能验收

- 用户可以从首页进入「以茶代酒」场景。
- 用户可以看到酒桌场景、角色头像和关系状态文字。
- 刚进入游戏主界面时，当前局势会以顶部滚动字幕播放一遍。
- 当前局势播放完成后会收起到左下角，hover 后完整展示。
- 状态反馈和小贴士默认隐藏在左下角信息抽屉组中，hover 后展示。
- 主界面不得出现预设选项按钮。
- 用户可以自由输入一句回应。
- 提交后系统能生成 AI NPC 反馈。
- 内部状态会根据用户输入变化，前端以隐藏抽屉中的文字状态和状态条变化呈现。
- 游戏至少支持两轮互动。
- 第二轮后可以进入结局页。
- 结局页展示复盘、知识卡、推荐话术。
- 用户可以点击「再玩一次」重置游戏。

### 15.2 场景验收

- 输入“开车不能喝，以茶代酒，表达敬意”应倾向触发「分寸拿捏」。
- 输入“那我喝一点吧”应显著提高 risk，并可能触发「失控风险」。
- 输入生硬拒绝应提高 boundary，但降低 face 和 atmosphere。
- 输入“王经理我能不喝吗”应增加 humanDebt 或降低 leaderRelation。
- 内部状态值必须始终保持在 0-100，前端不得直接展示这些数字。

### 15.3 UI 验收

- 桌面端主界面能完整展示酒桌、人物、顶部滚动字幕、左下角信息抽屉和输入区。
- 移动端头像、文本、信息抽屉、状态条、输入框不得互相遮挡。
- 文字不得横向溢出。
- 首页首屏可直接开始体验，不做纯营销落地页。
- 页面视觉应体现现代职场饭局，而不是古风或纯卡通问答。

### 15.4 技术验收

- `npm.cmd run build` 必须通过。
- 构建产物可运行。
- AI service 层与 UI 解耦。
- 场景内容可通过配置扩展。
- 前端代码中不得出现真实 AI API Key。

## 18. PRD 结论

《会来事儿》MVP 的关键不是做一个聊天框，而是做一个有场景、有角色、有压力、有反馈、有复盘的 AI 社交训练闭环。

首个场景「以茶代酒」需要让用户明确感受到：

```text
真正的会来事儿
不是无底线迎合
也不是生硬拒绝
而是在关系、面子、边界和风险之间
说出一句合适的话
```
